# 如何从 CubeMX 开始，按正确顺序独立完成这个 STM32 工程（代码施工版）

> 读者定位：零基础。  
> 目标：你不靠别人给代码，也能按顺序把工程写出来。  
> 说明：本手册只教你“先写什么、后写什么、为什么这样写”，不直接给最终项目代码。

---

## 0. 你要先建立的工作原则（先看完再动手）

1. 每次只做一个可验证的小目标。
2. 每新增一个文件，先写声明，再写实现，再编译。
3. 每完成一个阶段，必须过检查项，没过不能进入下一阶段。
4. 严禁一上来就写识别算法，先把“读数据、看数据、显示数据”走通。

---

## 1. 工程总施工顺序（总览）

1. 用 CubeMX 建基础工程并生成代码。
2. 先接入应用入口（`App_Init` / `App_Loop`）。
3. 先做 IMU 服务层（能稳定读一帧数据）。
4. 再做串口输出（先看到数据再谈算法）。
5. 再做 OLED 显示（验证状态可视化）。
6. 再做控制逻辑（先能控制一个 LED）。
7. 最后做识别引擎（阈值或 NanoEdge）。
8. 再做调参与验收。

---

## 2. CubeMX 详细配置（可直接照着点）

> 本章参数按当前工程代码基线对齐：  
> `MPU6050 -> I2C1(PB6/PB7)`，`OLED -> I2C2(PB10/PB11)`，`USART1=115200`，`USART2=9600`，`PD12=LED`。

### 2.1 新建工程与选型

| 项目 | 内容 |
|---|---|
| 目标 | 创建与你硬件一致的工程壳。 |
| 为什么先做这一步 | 芯片选错，后面引脚和外设全错。 |
| 具体做什么 | 打开 CubeMX -> `New Project` -> 搜索 `STM32F407VET6` -> 选择芯片 -> `Start Project`。 |
| 产出是什么 | 一个可保存的 `.ioc` 工程。 |
| 常见错误 | 选成别的封装或别的 STM32F4 子型号。 |
| 检查项 | 右上角芯片型号明确显示 `STM32F407VETx`。 |

### 2.2 RCC 与 SYS 基础设置

| 项目 | 内容 |
|---|---|
| 目标 | 让系统时钟和调试接口可用。 |
| 为什么先做这一步 | 时钟不稳定会导致串口/I2C全部异常。 |
| 具体做什么 | `Pinout & Configuration` 页面：`System Core -> RCC` 里把 `High Speed Clock(HSE)` 设为 `Crystal/Ceramic Resonator`；`System Core -> SYS` 把 `Debug` 设为 `Serial Wire`。 |
| 产出是什么 | HSE 外部时钟启用，SWD 可调试。 |
| 常见错误 | 忘开 SWD，后面下载调试不通。 |
| 检查项 | 引脚图里能看到 `PA13/PA14` 作为 SWD。 |

### 2.3 引脚分配（先分配再调参数）

| 模块 | 在 CubeMX 中操作 | 期望结果 |
|---|---|---|
| I2C1 | 点击 `PB6` 选 `I2C1_SCL`，点击 `PB7` 选 `I2C1_SDA` | 给 MPU6050 用 |
| I2C2 | 点击 `PB10` 选 `I2C2_SCL`，点击 `PB11` 选 `I2C2_SDA` | 给 OLED 用 |
| USART1 | `PA9=USART1_TX`，`PA10=USART1_RX` | 调试串口 |
| USART2 | `PA2=USART2_TX`，`PA3=USART2_RX` | 无线串口 |
| LED | `PD12` 设为 `GPIO_Output` | 业务动作输出 |

| 项目 | 内容 |
|---|---|
| 目标 | 先解决“资源冲突”。 |
| 为什么先做这一步 | 后调参数时，冲突会反复打断。 |
| 具体做什么 | 按上表逐个点引脚。出现冲突提示时，先取消冲突功能，再继续。 |
| 产出是什么 | 无冲突的引脚拓扑。 |
| 常见错误 | 只配了 I2C1，忘了 I2C2；只配 USART1，忘了 USART2。 |
| 检查项 | 引脚图上 5 组资源都已上色且无黄色警告。 |

### 2.4 时钟树（Clock Configuration）

| 项目 | 内容 |
|---|---|
| 目标 | 设成稳定主频，保证外设时序。 |
| 为什么先做这一步 | 主频错误会连带 I2C/UART 定时异常。 |
| 具体做什么 | 进入 `Clock Configuration`：`HSE=8MHz`；`PLLM=8`，`PLLN=336`，`PLLP=2`，`PLLQ=4`；确保 `SYSCLK=168MHz`，`AHB=168MHz`，`APB1=42MHz`，`APB2=84MHz`。 |
| 产出是什么 | 168MHz 主频配置。 |
| 常见错误 | PLL 参数填错导致红框或超频。 |
| 检查项 | 页面无红色超限提示；`SYSCLK` 显示 168MHz。 |

### 2.5 I2C1 参数（MPU6050）

| 项目 | 内容 |
|---|---|
| 目标 | 让 IMU 总线稳定可读。 |
| 为什么先做这一步 | 传感器数据是一切上层逻辑输入。 |
| 具体做什么 | `Connectivity -> I2C1`：`I2C mode`；`Clock Speed=100000`；`Addressing mode=7-bit`；其余默认。 |
| 产出是什么 | 代码中会生成 `MX_I2C1_Init()`。 |
| 常见错误 | 误设成 400k 在连线较差时不稳定。 |
| 检查项 | `i2c.c` 里 `hi2c1.Init.ClockSpeed` 为 `100000`。 |

### 2.6 I2C2 参数（OLED）

| 项目 | 内容 |
|---|---|
| 目标 | 让显示总线与传感器解耦。 |
| 为什么先做这一步 | 分总线可降低互相影响，调试更简单。 |
| 具体做什么 | `Connectivity -> I2C2`：同样设 `Clock Speed=100000`、`7-bit`。 |
| 产出是什么 | 代码中会生成 `MX_I2C2_Init()`。 |
| 常见错误 | 没启用 I2C2，却在代码中使用 `hi2c2`。 |
| 检查项 | `i2c.c` 里存在 `hi2c2` 和 `MX_I2C2_Init()`。 |

### 2.7 USART1 参数（调试口）

| 项目 | 内容 |
|---|---|
| 目标 | 提供主调试串口。 |
| 为什么先做这一步 | 没串口就失去最核心观察窗口。 |
| 具体做什么 | `Connectivity -> USART1`：`Mode=Asynchronous`，`BaudRate=115200`，`8 Data Bits`，`1 Stop Bit`，`Parity=None`，`Flow Control=None`。 |
| 产出是什么 | 代码中生成 `MX_USART1_UART_Init()`。 |
| 常见错误 | 上位机115200，代码却不是115200。 |
| 检查项 | `usart.c` 里 `huart1.Init.BaudRate = 115200`。 |

### 2.8 USART2 参数（无线口）

| 项目 | 内容 |
|---|---|
| 目标 | 提供第二路数据链路。 |
| 为什么先做这一步 | 便于调试与无线并行，不互相抢占。 |
| 具体做什么 | `Connectivity -> USART2`：同样异步模式，`BaudRate=9600`，其余 `8N1`。 |
| 产出是什么 | 代码中生成 `MX_USART2_UART_Init()`。 |
| 常见错误 | 误设115200导致无线模块端收不到。 |
| 检查项 | `usart.c` 里 `huart2.Init.BaudRate = 9600`。 |

### 2.9 GPIO（PD12 业务 LED）

| 项目 | 内容 |
|---|---|
| 目标 | 预留并确认业务输出脚。 |
| 为什么先做这一步 | 业务联动必须有明确输出点。 |
| 具体做什么 | 在引脚图把 `PD12` 设为 `GPIO_Output`；在 `GPIO` 参数中设为 `Push-Pull`、`No Pull`、`Low/Medium Speed`，默认输出 `Low`。 |
| 产出是什么 | `gpio.c` 中生成对应初始化。 |
| 常见错误 | 忘记把 PD12 配成输出。 |
| 检查项 | `MX_GPIO_Init()` 里能看到 `GPIO_PIN_12` 的配置。 |

### 2.10 NVIC 与 SysTick

| 项目 | 内容 |
|---|---|
| 目标 | 保证 `HAL_GetTick()` 正常节拍。 |
| 为什么先做这一步 | 你的调度全依赖毫秒 tick。 |
| 具体做什么 | 保持默认 SysTick 启用；不需要先改复杂中断优先级。 |
| 产出是什么 | 主循环可按 10/20/100ms 分时。 |
| 常见错误 | 误改系统时基导致周期失真。 |
| 检查项 | 运行时 `HAL_GetTick()` 单调递增。 |

### 2.11 Project Manager（工程生成选项）

| 项目 | 内容 |
|---|---|
| 目标 | 生成可持续迭代工程，不覆盖手写逻辑。 |
| 为什么先做这一步 | 选项错误会在二次生成时把你代码冲掉。 |
| 具体做什么 | `Project Manager` 页面：`Project Name=EdgeGestureHub`；`Toolchain=MDK-ARM`；勾选 `Generate peripheral initialization as pair of .c/.h files`；`Code Generator` 中勾选 `Keep User Code when re-generating`。 |
| 产出是什么 | 可重复生成且尽量不破坏手写代码的工程。 |
| 常见错误 | 没勾 `Keep User Code`，二次生成后用户代码丢失。 |
| 检查项 | 重新生成一次后，`USER CODE BEGIN/END` 区域内容仍在。 |

### 2.12 生成代码与首次验证

| 项目 | 内容 |
|---|---|
| 目标 | 验证 CubeMX 配置真实落地。 |
| 为什么先做这一步 | 纸面配置不等于代码正确。 |
| 具体做什么 | 点击 `GENERATE CODE`；进入 Keil 编译；查看 `i2c.c/usart.c/gpio.c/main.c` 是否含目标初始化函数。 |
| 产出是什么 | 第一版可编译工程。 |
| 常见错误 | 生成后没编译就继续写业务，导致问题来源不清。 |
| 检查项 | 至少一次编译通过，并能下载运行。 |

### 2.13 重新生成代码前的保护动作（很重要）

| 项目 | 内容 |
|---|---|
| 目标 | 防止重生代码把业务层破坏。 |
| 为什么先做这一步 | 新手最常见事故是“点一下生成，工作白做”。 |
| 具体做什么 | 业务代码尽量放在你自己新建的 `.c/.h`；Cube 自动文件只改 `USER CODE` 区域；生成前先提交一次版本。 |
| 产出是什么 | 可回滚、安全的生成流程。 |
| 常见错误 | 在自动生成区域写业务逻辑。 |
| 检查项 | 即使重生失败，也能从版本回到前一状态。 |

## 3. 文件创建与编写顺序（必须按这个顺序）

> 规则：每一行都是“先建 .h，再建 .c，再把它加入工程”。

| 顺序 | 先建头文件 | 再建实现文件 | 这一对文件负责什么 |
|---|---|---|---|
| 1 | `Core/Inc/app_config.h` | 无 | 全局参数（周期、阈值） |
| 2 | `Core/Inc/app_task.h` | `Core/Src/app_task.c` | 全工程调度入口 |
| 3 | `Core/Inc/imu_service.h` | `Core/Src/imu_service.c` | 统一 IMU 读取接口 |
| 4 | `Core/Inc/firewater.h` | `Core/Src/firewater.c` | 串口帧输出 |
| 5 | `Core/Inc/oled_service.h` | `Core/Src/oled_service.c` | OLED 业务显示封装 |
| 6 | `Core/Inc/control_logic.h` | `Core/Src/control_logic.c` | 业务动作执行 |
| 7 | `Core/Inc/gesture_engine.h` | `Core/Src/gesture_engine.c` | 识别引擎 |
| 8 | `Drivers/BSP/Inc/mpu6050.h` | `Drivers/BSP/Src/mpu6050.c` | 传感器驱动 |
| 9 | `Drivers/BSP/Inc/ssd1306.h` | `Drivers/BSP/Src/ssd1306.c` | 显示驱动 |

> 如果你的工程里这些文件已存在：不要重建，按本文顺序补齐内容。

## 4. 每个文件应该先写什么（函数顺序）

### 4.1 `app_config.h`（先写）
- 先写调度周期宏：采样、串口、OLED。
- 再写识别参数宏：学习锁时间、相似度阈值。
- 最后写注释，解释“调大/调小会发生什么”。

**检查项**：宏名称统一，后续所有 `.c` 能直接包含并使用。

### 4.2 `app_task.h/.c`（第二个写）
- 头文件先写：`App_Init`、`App_Loop`。
- 源文件顺序：静态缓存 -> `App_Init` -> `App_Loop`。

**检查项**：`main.c` 已调用 `App_Init/App_Loop`。

### 4.3 `imu_service.h/.c`（第三个写）
- 先写服务初始化函数。
- 再写读帧转发函数。

**检查项**：`IMU_ReadFrame` 成功返回可被上层识别。

### 4.4 `mpu6050.h/.c`（第四个写）
- 常量和寄存器定义 -> 底层读写 -> `Init` -> `ReadFrame`。

**检查项**：静止时数据稳定可解释。

### 4.5 `firewater.h/.c`（第五个写）
- 先写输出函数声明。
- 再写 CSV 格式化和串口发送。

**检查项**：输出字段固定 `ax,ay,az,gx,gy,gz,cls`。

### 4.6 `ssd1306` + `oled_service`（第六个写）
- 先驱动层，再服务层。

**检查项**：OLED 可稳定刷新状态文本。

### 4.7 `control_logic.h/.c`（第七个写）
- 初始化 -> 边沿检测 -> 冷却与门控。

**检查项**：不会连触发，回静止后可再次触发。

### 4.8 `gesture_engine.h/.c`（最后写）
- 初始化 -> 学习/检测 -> 稳定输出 -> 标签映射。

**检查项**：`LEARN/NORM/ANOM` 可稳定切换。

---

## 5. `main.c` 应该怎么改（只改必要位置）

1. 在用户代码区包含 `app_task.h`。
2. 在外设初始化后调用 `App_Init()`。
3. 在 `while(1)` 中只调用 `App_Loop()`。
4. 不要在 `main.c` 中写具体业务细节。

---

## 6. NanoEdge 阶段（必须在基础链路跑通后）

1. 先采“正常数据”和“异常数据”。
2. 再在 NanoEdge Studio 建工程并验证。
3. 再导出库与头文件并接入工程。
4. 在识别引擎中按顺序写：初始化 -> 学习分支 -> 检测分支 -> 状态映射。

**检查项**：上电先 `LEARN`，随后可出现 `NORM/ANOM`，串口与 OLED 一致。

---

## 7. 调试顺序（出问题时按这个查）

1. 先查供电与接线。
2. 再查 I2C 设备是否在线。
3. 再查串口参数和是否有数据。
4. 再查显示是否更新。
5. 最后才查识别与控制逻辑。

## 9. 每个阶段不该做的事

1. 需求阶段：不写代码。
2. 配置阶段：不改业务逻辑。
3. 驱动阶段：不引入 AI。
4. 调度阶段：不做大规模重构。
5. 验收阶段：不边测边改核心结构。

---

## 10. 给零基础同学的学习建议

1. 先学“怎么看数据”，再学“怎么让数据变聪明”。
2. 每天只攻一个模块，做完就记录“输入、输出、问题”。
3. 每通过一关就打标签保存版本。
4. 任何时候看不懂，先回到主链路：采样 -> 判定 -> 输出。

---

## 11. 你现在就可以开始的最小动作

1. 打开 CubeMX，按本手册第 2 章逐项设置到位。
2. 生成后先编译一次，再进入文件级编码施工。
3. 按第 3/4 章顺序，一对文件一对文件推进。

只要你严格按顺序做，这个工程你可以独立做完。
